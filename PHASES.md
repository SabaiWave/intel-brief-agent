# ScoutWork

AI-powered competitive intelligence briefs in minutes.

Multi-agent system built with Next.js, Anthropic SDK, and Tavily.
Each brief is generated by a team of specialist AI agents running in parallel,
synthesized into a structured competitive intelligence document.

---

## How It Works

1. User submits their company + up to 3 competitors + optional research focus
2. Orchestrator generates tailored instructions for each specialist agent
3. Agents run in parallel via `Promise.allSettled`:
   - **Research** — Tavily web search per company
   - **Positioning** — messaging, pricing signals, differentiators
   - **Competitor** — feature gaps, recent moves
   - **Content** — SEO signals, content strategy
4. Results stream to the UI in real time via Server-Sent Events (SSE)
5. Two-step synthesis: Analysis → Brief assembly
6. Output: structured Markdown competitive brief

---

## Tech Stack

- **Framework:** Next.js 14+ (App Router)
- **AI:** Anthropic Claude (`claude-sonnet-4-5`)
- **Search:** Tavily API
- **Styling:** Tailwind CSS
- **Deployment:** Vercel

## Phase 3 — Brief Quality & Prompt Iteration

```
Read CLAUDE.md before starting.

## Session Goal: Phase 3 — Brief Quality + Prompt Iteration

The backend and UI are working and deployed. The core product is the brief itself.
This phase is about making that brief genuinely good.

### What to do

1. **Run a real test**
   Generate a brief for a real company and 2-3 real competitors.
   Read the output critically. Identify the weakest section.

2. **Add token + cost logging**
   In each agent file (src/agents/), log the usage object from every
   Anthropic API response to the console:
   console.log(`[agent-name] tokens:`, response.usage)
   This tells us the real cost per brief at each depth setting.

3. **Identify the weakest agent**
   After running a real brief, determine which agent produces the
   thinnest or least useful output. Likely candidates: positioning or content
   (they reason over research rather than doing their own search).

4. **Improve the weakest prompt**
   Open the relevant file in src/prompts/ and improve it.
   Principles:
   - Be more specific about what to extract
   - Add explicit examples of good vs bad output
   - Instruct the agent to be opinionated, not hedged
   - Reference the actual company names in the prompt (already in context)

5. **Add per-agent timing to the UI**
   Each SSE status event already has a durationMs field in the type.
   Wire up actual timing in the orchestrator and display it in the
   agent status panel. Users waiting 60 seconds want to see progress.

6. **Tighten degraded output messaging**
   When an agent fails, the brief currently notes the gap generically.
   Make the degraded messaging specific:
   "Competitor agent failed — recent moves section based on research only."

### Definition of done
- Real brief tested and brief quality assessed
- Token usage logged per agent
- At least one prompt materially improved based on real output
- Agent timing visible in UI
- Degraded messaging specific and useful
```

## Phase 4 — Brief Persistence (Supabase)

```
Read CLAUDE.md before starting.

## Session Goal: Phase 4 — Brief Persistence

Currently briefs disappear on refresh. This phase adds persistence via Supabase
so users can return to their briefs. This is also the data foundation for auth
and monetization in later phases.

### Context
- Supabase project already exists (used in ecom-ops-agent)
- Add SUPABASE_URL and SUPABASE_ANON_KEY to .env.local
- Use @supabase/supabase-js

### Schema to create in Supabase

Table: briefs
- id: uuid (primary key, default gen_random_uuid())
- created_at: timestamptz (default now())
- your_company: text
- competitors: text[] 
- focus: text (nullable)
- depth: text
- brief_markdown: text        ← full brief output
- agent_statuses: jsonb       ← metadata.agentStatuses
- overall_confidence: text
- degraded: boolean

### What to build

1. **src/lib/supabase.ts**
   Supabase client singleton. Server-side only (uses service role key).

2. **src/lib/saveBrief.ts**
   Function that takes BriefingOutput and inserts to briefs table.
   Returns the saved brief id.

3. **Update app/api/brief/route.ts**
   After synthesis completes and before sending the 'complete' SSE event,
   call saveBrief() and include the brief id in the complete event:
   { type: 'complete', brief: BriefingOutput, briefId: string }

4. **app/brief/[id]/page.tsx**
   New page that fetches a brief by id from Supabase and renders it.
   Same brief renderer component as the main page.
   Shows "Brief not found" if id doesn't exist.

5. **Update main page UI**
   After a brief generates, show a shareable link:
   "Your brief is saved at scoutwork.com/brief/[id]"
   Copy-to-clipboard button.

### Definition of done
- Briefs saved to Supabase on generation
- /brief/[id] route renders saved brief correctly
- Shareable link shown after generation
- Brief persists across page refreshes
```

## Phase 6 — Stripe Paywall

```
Read CLAUDE.md before starting.

## Session Goal: Phase 6 — Stripe Paywall

Add a pay-per-brief model using Stripe. User pays before generating.
Keep it simple: one product, one price, one flow.

### Pricing decision (confirm before building)
Recommended starting point: $9 per brief (standard depth)
Deep depth: $19 per brief
Quick depth: free (lead gen, shows product value)

### Setup
1. Stripe account already exists (SBW)
2. Create product "ScoutWork Brief" in Stripe dashboard
   - Price 1: $9 one-time (standard)
   - Price 2: $19 one-time (deep)
3. Add to .env.local:
   STRIPE_SECRET_KEY=
   STRIPE_WEBHOOK_SECRET=
   NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY=
4. npm install stripe @stripe/stripe-js

### What to build

1. **src/lib/stripe.ts**
   Stripe client singleton.

2. **app/api/checkout/route.ts**
   POST endpoint. Receives { depth, briefRequestData }.
   Creates Stripe Checkout session with:
   - correct price based on depth
   - success_url: /brief/pending?session_id={CHECKOUT_SESSION_ID}
   - cancel_url: /
   - metadata: { briefRequestData: JSON.stringify(briefRequestData), userId }
   Returns { checkoutUrl }

3. **app/api/webhooks/stripe/route.ts**
   Stripe webhook handler.
   On checkout.session.completed:
   - Extract briefRequestData and userId from metadata
   - Run orchestrator (or enqueue it)
   - Save brief to Supabase with payment_status: 'paid'
   - Update session record so /brief/pending can poll for completion

4. **Supabase briefs table updates**
   Add columns:
   - stripe_session_id: text (nullable)
   - payment_status: text (default 'unpaid')

5. **app/brief/pending/page.tsx**
   Page shown after successful payment while brief generates.
   Polls Supabase every 3 seconds for the brief associated with session_id.
   When found, redirects to /brief/[id].
   Shows "Your brief is being generated..." with agent status if possible.

6. **Update main page**
   "Generate Brief" button now:
   - Quick depth: still fires directly (free)
   - Standard/Deep: calls /api/checkout, redirects to Stripe Checkout

### Note on webhook vs SSE
The SSE streaming approach works for synchronous generation (user waits on page).
With Stripe, generation happens after webhook fires — user is on /brief/pending.
Two options:
   A. Keep SSE for quick (free), use webhook + polling for paid tiers
   B. Move all generation to webhook + polling for consistency
Recommended: Option A for now — simpler, preserves the live demo experience for quick.

### Definition of done
- Quick depth: free, generates immediately with SSE (existing flow)
- Standard/Deep: redirects to Stripe Checkout
- After payment: brief generates via webhook, saves to Supabase
- /brief/pending polls and redirects when brief is ready
- Brief accessible at /brief/[id] after generation
```
